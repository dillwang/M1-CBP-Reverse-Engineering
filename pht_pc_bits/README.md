# pht_pc_bits

This folder contains the **PHT PC-bit alignment / aliasing test** for Apple M1 Firestorm branch-predictor reverse engineering.

The goal is to test which **program-counter bit positions** appear to matter for PHT-style conditional-branch predictor indexing or aliasing. The experiment builds a family of binaries named:

```text
firestorm_pht_align_a4
firestorm_pht_align_a5
...
firestorm_pht_align_a20
```

Each binary places two test-branch code regions, `.tb1` and `.tb2`, at addresses separated by exactly `2^A` bytes. Sweeping `A` changes one higher PC bit while keeping the lower address bits shared. If a specific `A` value produces noticeably different `rc5` counts, that is evidence that the corresponding PC bit participates in the predictor interaction being tested.

## Files

| File | Purpose |
|---|---|
| `build_align.sh` | Rebuilds `firestorm_pht_align_a<A>` binaries for a chosen range of `A`. |
| `link.ld` | Generated linker script from the most recent build. It places `.tb1` and `.tb2` at controlled addresses. |
| `firestorm_pht_align_a4` ... `firestorm_pht_align_a20` | Prebuilt AArch64 test binaries for each PC-bit separation value. |
| `scripts.txt` | Minimal command notes for rebuilding and running one measurement. |
| `plot_pht_pc_bits.py` | Plotting helper for manually collected `perf` data. |
| `pht_pc_bits_points_template.csv` | CSV template for manually entering measured `rc5` counts. |

The build script expects the source files `tb.c` and `tb.S` to exist in the same folder. They are not generated by the script.

## Build

Make the build script executable:

```bash
chmod +x build_align.sh
```

Build binaries for `A = 4..20`:

```bash
./build_align.sh 4 20
```

For each value of `A`, the script writes a temporary `link.ld`, computes `2^A`, and builds one binary:

```bash
gcc -O2 -fno-omit-frame-pointer tb.c tb.S -Wl,-T,link.ld -o firestorm_pht_align_a<A>
```

The linker script places `.tb1` at an aligned address and `.tb2` exactly `2^A` bytes after `.tb1`.

## Run one measurement

Example:

```bash
taskset -c 4 perf stat -e apple_firestorm_pmu/rc5/ ./firestorm_pht_align_a16 98 2000000 1
```

Arguments to the binary:

```text
./firestorm_pht_align_a<A> <dummyN> <iters> <seed>
```

| Argument | Meaning |
|---|---|
| `dummyN` | Number of dummy-chain links executed during the clear step. |
| `iters` | Number of test iterations. |
| `seed` | PRNG seed used by the test harness. |

The example above runs the `A=16` binary with `dummyN=98`, `2,000,000` iterations, and seed `1` while pinning execution to Firestorm core 4.

## Suggested manual sweep

A typical manual sweep is:

```bash
for A in $(seq 4 20); do
  echo "A=$A"
  taskset -c 4 perf stat -e apple_firestorm_pmu/rc5/ ./firestorm_pht_align_a${A} 98 2000000 1
 done
```

Record the `apple_firestorm_pmu/rc5/` count for each `A` into a CSV like this:

```csv
A,dummyN,iters,seed,rc5,notes
4,98,2000000,1,,
5,98,2000000,1,,
...
20,98,2000000,1,,
```

## Plot manually collected data

After filling in the `rc5` column:

```bash
python3 plot_pht_pc_bits.py pht_pc_bits_points.csv --outdir plots
```

If the CSV does not include an `iters` column, pass it explicitly:

```bash
python3 plot_pht_pc_bits.py pht_pc_bits_points.csv --iters 2000000 --outdir plots
```

Optional baseline subtraction:

```bash
python3 plot_pht_pc_bits.py pht_pc_bits_points.csv \
  --iters 2000000 \
  --baseline auto \
  --outdir plots
```

Outputs:

| Output | Meaning |
|---|---|
| `pht_pc_bits_normalized.csv` | Cleaned data with `rc5_per_iter` and optional `excess_rc5_per_iter`. |
| `pht_pc_bits_rc5_per_iter.png` | Main plot of `rc5 / iteration` versus `A`. |
| `pht_pc_bits_excess_per_iter.png` | Baseline-subtracted plot, generated when a baseline is provided. |
| `pht_pc_bits_top_positions.csv` | Highest-count `A` positions, useful for quickly checking likely PC-bit effects. |

## Interpretation

The sweep is intended to identify which PC bit positions contribute to the predictor aliasing behavior under this test setup.

- If two branches still interfere when separated by `2^A`, the changed bit may not fully distinguish the relevant predictor entry under this condition.
- If `rc5` changes sharply at a particular `A`, that bit position may affect the predictor index/tag/hash behavior.
- Results should be treated as evidence, not a complete proof, because PHT behavior can also depend on PHR state, target history, alignment, dummy-branch count, iteration count, and measurement noise.

## Notes

- This test is Apple M1 / Firestorm-specific because it uses the `apple_firestorm_pmu/rc5/` event.
- Use a fixed performance core, for example `taskset -c 4`, for more stable results.
- Re-run suspicious points multiple times before drawing conclusions.
- Keep `dummyN`, `iters`, and `seed` constant when comparing across `A` values.
- `perf stat` may require kernel support for the Apple Firestorm PMU event.
