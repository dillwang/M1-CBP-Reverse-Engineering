# `phr_target_indirect`

This folder contains an Apple M1 / Firestorm branch-predictor microbenchmark for testing whether **indirect branch targets** influence the branch-history state used by later conditional-branch prediction.

The experiment varies the distance between two indirect-call targets, `L0` and `L1`, and then measures whether the later probe branch becomes harder to predict. In the intended interpretation, a rise in `rc5` mispredictions suggests that the selected indirect target address is being incorporated into the relevant history/index state.

## High-level idea

Each loop iteration does roughly this:

```text
k = random bit
indirect call target = (k ? L1 : L0)
call selected target through one fixed indirect-call site
execute N dummy unconditional branches
execute a conditional probe branch based on k
```

`L0` and `L1` are tiny functions that only return. The linker script generated by `build_bits.sh` places them at a controlled distance of `2^bit` bytes apart. Sweeping `bit` changes which target-address bit differs between `L0` and `L1`.

If changing a particular target-address bit changes the later probe-branch misprediction count, that bit is a candidate for affecting the history or index state used by the conditional predictor.

## Important files

| File | Purpose |
| --- | --- |
| `tb.c` | C driver. Parses `ndummy`, `iters`, and optional `seed`, then calls `run_target_ld`. |
| `tb.S` | Main benchmark loop. Generates random `k`, performs the indirect call to `L0`/`L1`, runs the dummy branch chain, and executes the final conditional probe branch. |
| `targets.S` | Defines `L0` and `L1` in separate sections `.tgt0` and `.tgt1`. Both targets simply `ret`. |
| `build_bits.sh` | Builds one binary per target-distance bit by generating a temporary `link.ld`. |
| `sweep_target.sh` | Runs all built binaries over a dummy-branch-count sweep and records `rc5` results in `results.csv`. |
| `link.ld` | Generated by `build_bits.sh`; usually overwritten during builds. |

## Build

Make the scripts executable:

```bash
chmod +x build_bits.sh sweep_target.sh
```

Build binaries for a range of target-distance bits:

```bash
./build_bits.sh 2 31
```

This produces binaries named:

```text
firestorm_target_ld_b2
firestorm_target_ld_b3
...
firestorm_target_ld_b31
```

For each bit value `b`, the generated linker script places:

```text
L1 address = L0 address + 2^b bytes
```

The script intentionally inserts the generated section-placement fragment after `.eh_frame` while keeping the default program-header / load layout.

## Manual run

Each binary has this runtime interface:

```bash
./firestorm_target_ld_b<bit> <ndummy> <iters> [seed]
```

Example:

```bash
./firestorm_target_ld_b16 100 2000000 1
```

Measure Firestorm branch mispredictions with `perf` on core 4:

```bash
taskset -c 4 perf stat -e apple_firestorm_pmu/rc5/ \
    ./firestorm_target_ld_b16 100 2000000 1
```

Arguments:

| Argument | Meaning |
| --- | --- |
| `bit` | Target-distance bit. `L0` and `L1` differ by `2^bit` bytes. |
| `ndummy` | Number of dummy unconditional branches before the conditional probe. |
| `iters` | Number of benchmark loop iterations. |
| `seed` | Optional xorshift seed; default is `1`. |

## Full sweep

After building all binaries expected by the sweep script, run:

```bash
./sweep_target.sh
```

Default sweep parameters inside the script:

```text
bit:    2..31
ndummy: 65..100
iters:  2,000,000
seed:   1
core:   4
event:  apple_firestorm_pmu/rc5/
```

The sweep writes:

```text
results.csv
```

with columns:

| Column | Meaning |
| --- | --- |
| `bit` | Target-distance bit used for the binary. |
| `ndummy` | Dummy-branch count passed to the benchmark. |
| `ndummy_effective` | `ndummy + 2`, used by the script to account for fixed extra branch effects. |
| `iters` | Loop iteration count. |
| `rc5` | Raw Firestorm `rc5` branch-misprediction count. |
| `miss_per_iter` | `rc5 / iters`. |
| `seconds` | Runtime reported by `perf`, if parsed successfully. |

## Interpreting results

The main quantity to inspect is `rc5` or `miss_per_iter` as a function of:

```text
bit × ndummy_effective
```

A useful interpretation rule is:

```text
flat / baseline rc5       => selected target bit probably does not matter in this setup
higher rc5 at some bit    => selected target bit may affect the relevant history/index state
shift with ndummy         => effect depends on where the injected target information lands in history
```

This experiment does not prove the exact internal predictor formula by itself. It provides evidence that a particular indirect-target address bit can be observed indirectly through later conditional-branch prediction behavior.

## Notes and cautions

- This benchmark is Apple M1 / Firestorm-specific because it uses the PMU event `apple_firestorm_pmu/rc5/`.
- Use `taskset -c 4` or another fixed Firestorm performance core to reduce noise.
- `build_bits.sh` overwrites `link.ld` during the build sweep.
- `sweep_target.sh` expects all binaries for `bit=2..31` to already exist.
- `tb.c` allows `ndummy` in `0..160`; the default sweep uses `65..100`.
- The targets `L0` and `L1` only contain `ret`, so the intended changing variable is the target address, not target-body behavior.
- High misprediction counts should be interpreted comparatively against neighboring bit values and control-style runs, not as an absolute proof of one predictor structure.
